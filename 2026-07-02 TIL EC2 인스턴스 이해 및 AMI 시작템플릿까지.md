## 1. AWS 계정 기초 개념

## 2. AWS CLI 자격증명 설정

### 자격증명이 없을 때 에러
```bash
aws sts get-caller-identity
# aws: [ERROR]: An error occurred (NoCredentials): Unable to locate credentials.
```

### aws configure로 설정하기
```bash
aws configure
# AWS Access Key ID: [액세스 키 입력]
# AWS Secret Access Key: [시크릿 키 입력]
# Default region name: ap-northeast-2
# Default output format: json
```

### 설정 파일 위치 확인
```bash
ls .aws          # cli  config  credentials
cat .aws/config  # region, output 확인
```

### 자격증명 확인
```bash
aws sts get-caller-identity
# UserId, Account, Arn 정보 출력
```

## 3. EC2 인스턴스 개념

> "논리적인 컴퓨터를 하나 더 놔둔 셈"

### 주요 구성 요소

| 구성 요소 | 설명 |
|---|---|
| 인스턴스 유형 | CPU와 메모리를 하나로 묶어둔 것. 일반적으로 t3 타입 사용 |
| EBS | 저장 공간 (하드디스크 역할) |
| AMI | 운영체제 (어떤 OS를 쓸 것인가) |
| 네트워크 | 어디에 위치시킬 것인가 (VPC, 서브넷) |
| 보안 그룹 | 내 컴퓨터를 무슨 용도로 쓸 것인가 (방화벽 역할) |

### IP와 포트
```
인터넷 → IGW(인터넷 게이트웨이) → EC2(IP)
IP는 대문, 그 안으로 들어가기 위해서는 포트 번호가 필요
- 웹 서비스: 80 (HTTP), 443 (HTTPS)
- SSH: 22
```

### 보안 그룹 규칙
- **인바운드**: 기본적으로 다 닫혀있음 → 허용할 규칙을 직접 추가해야 함
- **아웃바운드**: 기본적으로 다 열려있음
- **TCP 프로토콜은 양방향**

> 💡 문제가 생긴다면 대부분 보안 그룹 문제임

## 4. EC2 인스턴스 콘솔로 생성하기

> AWS 전체 메뉴 → 컴퓨팅 → EC2 → 인스턴스(리전 확인) → 인스턴스 시작

### 설정 순서
1. **이름**: `std03-ex-ec2` (네임 태그 — 태그는 수정 가능)
2. **AMI**: Ubuntu 24.04 64비트 x86 선택 (한 단계 낮은 버전이 더 안정화되어 있음)
3. **인스턴스 유형**: t3.micro (CPU, 메모리, 비용 고려)
4. **키페어**: std03 .pem → 사용자 > .ssh에 저장
5. **네트워크**: 기본 VPC는 리전당 하나씩. 퍼블릭 서브넷 선택
6. **퍼블릭 IP 자동 할당**: 활성화
7. **보안 그룹 생성**: `std03-ssh-sg` / 인바운드 SSH 추가
   - (실수: `std-ssh-sg`로 만들었으나 `std03-ssh-sg`였어야 함)
8. **스토리지**: 20GB 권장
   - 서비스를 깔면 로그와 데이터가 생성되기 때문에 단순히 설치 용량만 기준으로 하면 안 됨
   - 추후 문제 발생 시 로그를 보고 판단하기 쉽게 저장 공간을 따로 두는 것을 추천

### VPC 퍼블릭 IP 자동 할당 활성화
> VPC → 서브넷 → 작업 → 서브넷 설정 편집 → 퍼블릭 IPv4 주소 자동 할당 활성화

## 5. Nginx 설치 및 보안 그룹 설정

### EC2 접속 후 Nginx 설치
```bash
sudo apt update              # 설치 가능한 패키지 목록 업데이트
sudo apt install nginx -y    # Nginx 설치
sudo systemctl enable nginx  # 부팅 시 자동 실행 등록
sudo systemctl start nginx   # Nginx 실행
sudo systemctl status nginx  # 상태 확인

ip addr                      # 이 인스턴스에 부여된 랜카드 정보와 IP 주소 전체 출력
curl localhost               # 로컬에서 Nginx 응답 확인
```

### 왜 퍼블릭 IP로 접속이 안 되냐?
인바운드는 기본적으로 외부에서 접속하는 걸 다 막아두기 때문. HTTP 보안 그룹을 만들어서 EC2에 추가해야 함.

### HTTP 보안 그룹 만들기
> 네트워크 및 보안 → 보안 그룹 → 보안 그룹 생성
> - 이름: `std03-http-sg` / 설명: http
> - 인바운드 규칙 추가: HTTP, Anywhere-IPv4
> - 태그: Name=`std03-http-sg`, Owner=`std03`

> EC2 인스턴스 → 작업 → 보안 → 보안 그룹 변경 → 새로 만든 보안 그룹 추가 → 저장

### 트래픽 흐름
```
USER → IP → 보안 그룹(80, 443) → EC2(Nginx)
```

## 6. 사용자 데이터(User Data)로 자동 설치

### 사용자 데이터란?
인스턴스가 **처음 시작될 때 딱 한 번** 자동으로 실행되는 스크립트.
중지 후 재시작하면 사용자 데이터는 다시 실행되지 않음.

### 인스턴스 생성 시 사용자 데이터 입력
> 인스턴스 생성 → 고급 세부 정보 → 사용자 데이터에 입력

```bash
#!/bin/bash
apt update -y
apt install -y nginx
systemctl enable nginx
systemctl start nginx
```

## 7. AMI(Amazon Machine Image) 생성

기존 인스턴스를 그대로 복제해서 새 인스턴스를 빠르게 만들 수 있음.

> 인스턴스 선택 → 작업 → 이미지 및 템플릿 → 이미지 생성
> - 이름: `std03-nginx-ami`
> - 인스턴스 재부팅: 항상 체크
> - 태그 추가: Name=`std03-nginx-ami`
> - 이미지 생성 완료 후 → AMI에서 인스턴스 시작

> (실수: `std-nginx-ami`로 만들었으나 `std03-nginx-ami`였어야 함)

### 이미지 종류 두 가지
| 종류 | 설명 |
|---|---|
| AMI | AWS에서 제공하는 기본 이미지 또는 내가 만든 커스텀 이미지 |
| Docker 이미지 | 여러 계층으로 이루어진 파일. 필요 없는 건 제거한 형태 |

## 8. 시작 템플릿(Launch Template)

> "구성을 해주는 것. 원칙을 지정해주는 것. 미리 만들어진 양식."

### 시작 템플릿 만들기
> 인스턴스 → 시작 템플릿 → 시작 템플릿 생성
> - 이름: `std03-nginx-tmp`, 설명: `temp test`
> - 오토 스케일링 지침 체크 (중요)
> - AMI: 내 AMI 선택 (`std03-nginx-ami`)
> - 네트워크 인터페이스: 새 인터페이스
> - 리소스 태그: 리소스 유형(인스턴스, 볼륨), Name=`std03-temp-instance`

### 템플릿 버전 관리
> 시작 템플릿 → 작업 → 템플릿 새 버전 생성 → 템플릿 버전 설명 입력 (중요) → 생성

> 💡 오토 스케일링에서 사용할 경우 서브넷은 시작 템플릿에 포함하지 않는 것을 권장

## 9. AWS CLI로 EC2 인스턴스 생성

### CLI 명령어 기본 규칙
```
aws 명령어는 무조건 'aws'로 시작
-  → 한 글자 옵션 (예: -o)
-- → 단어 옵션 (예: --output)
```

### 데이터 구조 참고
```
리스트: [값, 값, 값]
맵: {"key명": 값}
태그 형식: [{"Key": "키값", "Value": "값"}, {...}]
```

### 최종 성공 명령어
```bash
aws ec2 run-instances \
  --image-id ami-0fa4188731e6180ca \
  --count 1 \
  --instance-type t3.micro \
  --key-name std03-key \
  --subnet-id subnet-02cc50181fd23c713 \
  --associate-public-ip-address \
  --security-group-ids sg-05508a5736464eb2d \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=std03-instance},{Key=Owner,Value=std03}]'
```

### 인스턴스 종료
```bash
aws ec2 terminate-instances --instance-ids [인스턴스 ID]
```

## 오늘 막혔던 부분 (실수 기록)

1. **보안 그룹 이름 실수**: `std-ssh-sg` → 원래 `std03-ssh-sg`였어야 함. 이름 앞에 `std03-` 붙이는 것 습관화.
2. **AMI 이름 실수**: `std-nginx-ami` → `std03-nginx-ami`였어야 함.
3. **CLI 옵션 앞 `--` 빠뜨림**: `image-id` → `--image-id`
4. **키페어 이름 틀림**: `std-03` → `std03-key` (콘솔에서 정확한 이름 확인 필수)
5. **AMI가 리전에 없음**: AMI는 리전마다 다름. 현재 리전(ap-northeast-2) 기준 ID 사용해야 함.
6. **서브넷과 보안 그룹이 다른 VPC**: 서브넷과 보안 그룹은 반드시 같은 VPC 안에 있어야 함.

## 다음에 헷갈리지 않으려면
- 리소스 이름은 항상 `std03-` prefix 붙이기
- AMI ID, 서브넷 ID, 보안 그룹 ID는 실행 전 콘솔에서 정확히 확인
- 서브넷과 보안 그룹은 반드시 같은 VPC
- 사용자 데이터는 최초 한 번만 실행됨 (재시작해도 다시 실행 안 됨)
- 시작 템플릿에 오토 스케일링 쓸 경우 서브넷 포함하지 말 것
- Access Key는 절대 외부 공유/깃허브 업로드 금지
